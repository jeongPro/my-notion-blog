# ROADMAP: Notion CMS 블로그 개발 로드맵

> PRD 기반으로 작성된 단계별 개발 계획서

---

## 개발 순서 개요

```
1. 프로젝트 초기 설정 → 2. 공통 모듈 개발 → 3. 핵심 기능 개발 → 4. 추가 기능 개발 → 5. 최적화 및 배포
```

---

## Phase 1: 프로젝트 초기 설정 (골격 구축)

### 왜 이 단계가 먼저인가?
프로젝트의 기반이 되는 설정 없이는 어떤 코드도 작성할 수 없다. 개발 환경, 폴더 구조, 의존성이 먼저 갖춰져야 이후 모든 작업이 원활하게 진행된다.

### 작업 목록

| 순서 | 작업 | 설명 |
|------|------|------|
| 1-1 | Next.js 15 프로젝트 생성 | App Router 기반 프로젝트 초기화 |
| 1-2 | TypeScript 설정 | tsconfig.json 최적화 |
| 1-3 | Tailwind CSS 설정 | 스타일링 기반 구축 |
| 1-4 | shadcn/ui 초기화 | components.json 설정, cn 유틸리티 |
| 1-5 | 폴더 구조 생성 | app/, components/, lib/, types/ 등 |
| 1-6 | 환경 변수 설정 | .env.example, .env.local 템플릿 |
| 1-7 | ESLint/Prettier 설정 | 코드 품질 도구 구성 |
| 1-8 | Git 초기 설정 | .gitignore 확인, 초기 커밋 |

### 완료 기준
- [ ] `npm run dev` 실행 시 에러 없이 로컬 서버 구동
- [ ] `npm run build` 성공
- [ ] 모든 폴더 구조가 PRD 명세대로 생성됨
- [ ] 환경 변수 템플릿 파일 존재

---

## Phase 2: 공통 모듈/컴포넌트 개발

### 왜 이 단계가 두 번째인가?
핵심 기능을 개발하기 전에 재사용될 공통 요소들을 먼저 만들어야 한다. 타입 정의, API 클라이언트, 레이아웃 컴포넌트는 여러 페이지에서 공통으로 사용되므로 먼저 구축해야 중복 코드를 방지할 수 있다.

### 작업 목록

| 순서 | 작업 | 설명 |
|------|------|------|
| 2-1 | 타입 정의 | Post, Block 인터페이스 (types/index.ts) |
| 2-2 | Notion API 클라이언트 설정 | @notionhq/client 설치 및 초기화 |
| 2-3 | Notion API 유틸리티 함수 | getPosts, getPostBySlug 등 (lib/notion.ts) |
| 2-4 | 공통 유틸리티 함수 | 날짜 포맷, 슬러그 변환 등 (lib/utils.ts) |
| 2-5 | Header 컴포넌트 | 로고, 네비게이션 |
| 2-6 | Footer 컴포넌트 | 저작권, 링크 |
| 2-7 | 루트 레이아웃 구성 | app/layout.tsx에 Header/Footer 적용 |

### 완료 기준
- [ ] Notion API 연결 테스트 성공 (데이터 fetch 확인)
- [ ] 타입 정의 완료 및 타입 에러 없음
- [ ] Header/Footer가 모든 페이지에 표시됨
- [ ] 공통 유틸리티 함수 동작 확인

---

## Phase 3: 핵심 기능 개발

### 왜 이 단계가 세 번째인가?
공통 모듈이 준비되었으므로 이제 사용자에게 가치를 제공하는 핵심 기능을 구현한다. MVP에 포함된 필수 기능들을 우선 개발하여 블로그의 기본 동작을 완성한다.

### 작업 목록

| 순서 | 작업 | 설명 |
|------|------|------|
| 3-1 | PostCard 컴포넌트 | 글 미리보기 카드 UI |
| 3-2 | PostList 컴포넌트 | 글 목록 표시 |
| 3-3 | 홈 페이지 구현 | 히어로 섹션 + 최근 글 목록 (app/page.tsx) |
| 3-4 | Notion 블록 렌더러 | 텍스트, 헤딩, 리스트 블록 |
| 3-5 | 코드 블록 렌더러 | 신택스 하이라이팅 적용 |
| 3-6 | 이미지 블록 렌더러 | Next.js Image 최적화 |
| 3-7 | PostContent 컴포넌트 | 블록 렌더러 통합 |
| 3-8 | 글 상세 페이지 | app/posts/[slug]/page.tsx |
| 3-9 | 카테고리 페이지 | app/category/[category]/page.tsx |
| 3-10 | 404 페이지 | 존재하지 않는 페이지 처리 |

### 완료 기준
- [ ] 홈 페이지에서 Notion 글 목록 표시
- [ ] 글 클릭 시 상세 페이지로 이동
- [ ] 상세 페이지에서 모든 Notion 블록 정상 렌더링
- [ ] 카테고리별 필터링 동작
- [ ] 코드 블록 하이라이팅 적용

---

## Phase 4: 추가 기능 개발

### 왜 이 단계가 네 번째인가?
핵심 기능이 완성된 후에 사용자 경험을 향상시키는 부가 기능을 추가한다. 이 기능들은 블로그의 기본 동작에 영향을 주지 않으므로 후순위로 개발한다.

### 작업 목록

| 순서 | 작업 | 설명 | 우선순위 |
|------|------|------|----------|
| 4-1 | 검색 기능 | 제목 기반 실시간 검색 | 높음 |
| 4-2 | 페이지네이션 | 글 목록 페이지 분할 | 높음 |
| 4-3 | 이전/다음 글 네비게이션 | 상세 페이지 하단 | 중간 |
| 4-4 | 태그 표시 및 필터링 | 태그 클릭 시 필터 | 중간 |
| 4-5 | 다크 모드 | 테마 토글 기능 | 낮음 |
| 4-6 | 목차(TOC) 자동 생성 | 헤딩 기반 네비게이션 | 낮음 |

### 완료 기준
- [ ] 검색 입력 시 실시간 결과 표시
- [ ] 페이지네이션 또는 무한 스크롤 동작
- [ ] 추가 기능들이 기존 기능과 충돌 없이 동작

---

## Phase 5: 최적화 및 배포

### 왜 이 단계가 마지막인가?
기능 개발이 완료된 후에 성능 최적화와 배포를 진행한다. 불완전한 기능 상태에서 최적화하면 이후 변경 시 다시 작업해야 하므로 비효율적이다.

### 작업 목록

| 순서 | 작업 | 설명 |
|------|------|------|
| 5-1 | 반응형 디자인 검증 | 모바일/태블릿/데스크톱 테스트 |
| 5-2 | 이미지 최적화 | Next.js Image 컴포넌트 활용 |
| 5-3 | 캐싱 전략 적용 | ISR 또는 revalidate 설정 |
| 5-4 | SEO 메타 태그 | 각 페이지별 동적 메타 태그 |
| 5-5 | OG 이미지 설정 | 소셜 공유용 이미지 |
| 5-6 | Lighthouse 점검 | 성능/접근성/SEO 점수 확인 |
| 5-7 | Vercel 배포 설정 | 프로젝트 연결, 환경 변수 |
| 5-8 | 프로덕션 배포 | 최종 배포 및 테스트 |
| 5-9 | 도메인 연결 (선택) | 커스텀 도메인 설정 |

### 완료 기준
- [ ] Lighthouse 점수 90점 이상 (Performance, SEO)
- [ ] LCP < 2.5초
- [ ] 모든 기기에서 반응형 정상 동작
- [ ] Vercel 배포 성공 및 프로덕션 URL 접근 가능

## 참고사항

- 각 Phase는 이전 Phase 완료 후 진행
- Phase 3의 세부 작업은 병렬 진행 가능
- Phase 4의 작업은 우선순위에 따라 선택적 진행 가능
- 예상 시간은 개발자 숙련도에 따라 달라질 수 있음
